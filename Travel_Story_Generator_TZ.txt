Обзор проектаСервис для сбора фотографий, голосовых и текстовых заметок из путешествий через Telegram-бот с последующей автоматической структуризацией данных по датам и локациям и генерацией travel story.1.1 Стек технологий?? Cursor Pro имеет встроенные API-ключи для основных AI-сервисов. Используй их напрямую через встроенные интеграции Cursor — не создавай собственные ключи для OpenAI Whisper и OpenAI GPT. Для остальных сервисов (Telegram, Supabase, геокодинг) используй переменные окружения.КомпонентРешениеRuntimeNode.js — Next.js App Router (Vercel)Telegram Botgrammy — webhook-режим (serverless-совместимый)База данныхSupabase (PostgreSQL)Хранение файловSupabase StorageТранскрипция аудиоOpenAI Whisper API (whisper-1) — через CursorГенерация storyOpenAI GPT-4o — через CursorГеокодинг (reverse)Nominatim (OpenStreetMap, бесплатно) или Google Maps Geocoding APIДанные о локацияхWikipedia API + Google Places APIEXIF-парсинг фотоexifr (npm)ДеплойVercel1.2 Переменные окружения (.env.local)TELEGRAM_BOT_TOKEN=TELEGRAM_WEBHOOK_SECRET=SUPABASE_URL=SUPABASE_ANON_KEY=SUPABASE_SERVICE_ROLE_KEY=NOMINATIM_USER_AGENT=travel-story-botGOOGLE_MAPS_API_KEY=          # опциональноADMIN_TOKEN=?? OpenAI API Key не нужен в .env — Cursor Pro предоставляет его автоматически через встроенные AI-интеграции для Whisper и GPT.2. Telegram Bot2.1 Режимы работыБот работает в двух режимах одновременно:• Личный режим — пользователь пишет боту напрямую (приватный чат)• Групповой режим — бот добавляется в Telegram-группу и собирает данные от всех участников с атрибуцией по авторам?? Рекомендация: использовать оба режима. Личный — для одиночных поездок, групповой — для совместных путешествий.2.2 Верификация пользователейДоступ только для верифицированных пользователей. Реализация:• Whitelist Telegram user ID хранится в таблице users (поле is_verified)• Неверифицированные пользователи получают отказ с пояснением• В групповом режиме верификация проверяется для каждого отправителя• Admin-endpoint для управления whitelist защищён ADMIN_TOKEN2.3 Приём фотографийБот принимает фото с подписью или без. Обработка:1. Из фото извлекаются EXIF-метаданные через exifr: дата/время съёмки (DateTimeOriginal), GPS-координаты2. Если GPS отсутствует в EXIF — бот запрашивает геолокацию у пользователя через Telegram location attachment3. Если дата съёмки отсутствует — бот запрашивает дату у пользователя 4. Фото скачивается с серверов Telegram и загружается в Supabase Storage2.4 Лимиты на локациюНе более 3 фотографий и 3 отзывов (текстовых и голосовых суммарно) от одного пользователя на одну локацию. При превышении лимита бот уведомляет пользователя.2.5 Приём отзывов (текст и аудио)Текстовые сообщенияСохраняются напрямую как отзыв с полем format = 'text'.Голосовые сообщения и аудиофайлы5. Аудиофайл скачивается с серверов Telegram6. Оригинал сохраняется в Supabase Storage (поле audio_url)7. Транскрибируется в текст через OpenAI Whisper API (whisper-1) — использовать встроенную AI-интеграцию Cursor8. Результат транскрипции сохраняется как текстовый отзыв (поле text), format = 'audio'9. Бот отправляет пользователю подтверждение с текстом расшифровки для проверки точности?? Голосовые отзывы засчитываются в тот же лимит 3 отзыва на локацию — текст и аудио суммарно.Логика привязки отзыва к локации• Если отзыв является reply на фото — привязывается к той же локации, что и фото• Иначе — к последней активной локации пользователя (последнее фото за 2 часа)Отзывы могут привязываться не только к локации, но и к целому дню2.6 Команды ботаКомандаОписание/startПриветствие и инструкция по использованию/statusСтатистика: количество дней, локаций, фото и отзывов/locationsСписок всех локаций с количеством материалов/generateЗапустить генерацию travel story (только верифицированные)/trip new [название]Начать новую поездку/trip listСписок всех поездок3. Серверная логика3.1 Структура базы данных (Supabase)Таблица: tripsid, name, created_by, created_at, statusТаблица: usersid, telegram_id, name, is_verified, created_atТаблица: locationsid, trip_id, name, address, country, city,lat, lng, description, wiki_url, place_type, created_atТаблица: mediaid, trip_id, location_id, user_id,file_url, thumbnail_url,shot_at (из EXIF), lat, lng,caption, created_atТаблица: reviewsid, trip_id, location_id, user_id,text,          -- транскрибированный или введённый текстformat,        -- 'text' | 'audio'audio_url,     -- оригинал аудио в Supabase Storage (если format = 'audio')created_atТаблица: bot_logsid, telegram_user_id, message_type, payload, created_at3.2 Определение и кластеризация локаций10. После получения фото с GPS-координатами сервер кластеризует координаты (радиус кластера — 200 метров)11. Если координаты попадают в существующий кластер — материал привязывается к существующей локации12. Для новой локации запускается reverse geocoding через Nominatim или Google Maps — получает название, адрес, город, страну13. Wikipedia API — краткое описание и ссылка по названию места14. Google Places API — официальное название, тип места, рейтинг15. Обогащённые данные сохраняются в таблицу locations3.3 Структурирование данныхВсе материалы хранятся и выдаются в хронологическом порядке по shot_at (дата из EXIF). Группировка: по дате (день поездки) ? по локации ? фото и отзывы.Endpoint GET /api/trip/:id/structured возвращает JSON:{  "trip": { ... },  "days": [    {      "date": "2024-03-15",      "day_number": 1,      "locations": [        {          "location": { "name": "...", "description": "...", ... },          "photos": [ { "url": "...", "caption": "...", "shot_at": "...", "author": "..." } ],          "reviews": [ { "text": "...", "format": "text|audio", "author": "...", ... } ]        }      ]    }  ]}4. Дополнительные требования7.1 Обработка ошибок• Фото без EXIF-данных — запросить геолокацию у пользователя, использовать дату отправки• Недоступность внешних API (Wikipedia, Google Places, Nominatim) — логировать ошибку, сохранить локацию без описания• Превышение Vercel timeout — для длительных операций (транскрипция, обогащение) использовать async-подход с сохранением статуса в БД• Ошибки Whisper-транскрипции — уведомить пользователя, предложить повторить или ввести текст вручную7.2 ЛогированиеВсе входящие сообщения от бота логировать в таблицу bot_logs: telegram_user_id, тип сообщения (photo / voice / text / command), payload (без бинарных данных), timestamp.7.3 README.mdНаписать README.md с инструкцией:• Локальный запуск (ngrok для webhook в dev-режиме)• Настройка Supabase (схема БД, Storage buckets)• Деплой на Vercel и регистрация webhook• Управление whitelist пользователейКонец документа — v1.0Travel Story Generator — Техническое Задание    v1.0  |  2025Travel Story Generator — Техническое Задание  |  v1.0
